# 0706 通用脚手架深度诊断与修复

## 背景

以资深架构师视角对脚手架做**开放式深度诊断**（不预设问题），目标基准：让派生项目"壳成本≈0、业务有范可循、结构高清晰、演进可持续"。诊断阶段派探索 agent 分头通读文档/组件层、调研"模板仓库回流"业界做法、再派对抗性 subagent 复审（推翻初稿 3 处证据 + 2 处排序、补 3 条盲区），产出 10 条正式发现的报告（含 HTML 可视化）。本次逐条修复，8 个提交，全绿。

## 逐条修复（按 ROI 顺序）

### F1/F3 文档入口收敛
- 问题：CLAUDE.md（每次 AI 会话必读、优先级最高）子系统清单教横切结构，与 AGENTS 纵切铁律直接冲突；`frontend-architecture-guide.md` 是孤儿文档、核心样例指向已删的 `pages/users/`；README 仍是"原型交接稿"（推荐 antd+react-router，与实际栈矛盾）；`architecture.md` 大小写 git/磁盘不一致。
- 过程：**CLAUDE.md 改为 AGENTS.md 软链接**（洋哥提议，先 readlink/git mode 核实"当时其实是独立文件不是软链接"才动手），单一真相源；删 guide；README 重写为工程入口（quick start + 文档地图），原型稿移 `docs/prototype-handoff.md`；新增 `docs/NEW-PROJECT.md` 派生清单；architecture 加"核心 vs 示例业务"边界标注；文件名统一小写；spec/baseline 加历史标注。
- 结论：规则不再有两份漂移；文档入口不再"越靠前越过时"。

### F10 仓库卫生

### F5 dev 画廊生产门
- 问题：`/dev/theme-states` 无 staticData、无环境门，任何登录用户直连 URL 即可访问内部组件画廊，且进生产 bundle。
- 过程：加 `beforeLoad` 环境门（非 dev 且未开 VITE_ENABLE_VISUAL_DEBUG 则 `throw notFound()`）；加守卫测试锁"dev 路由必须有门"；config 补 isDev/enableVisualDebug 三态单测（替代受阻的生产 e2e）。

### F2 壳的全局错误兜底
- 问题：全仓库无 `defaultErrorComponent`，读 TanStack `Match.js` 确认：渲染期错误的 CatchBoundary 退化为 SafeFragment 不捕获 → 冒泡到 React 根 → **整树卸载白屏**（不是"丑错误框"，更严重）。
- 过程：createRouter 配 `defaultErrorComponent`（按 Contract/Auth/Http/Biz 错误类型给文案 + 重试）；ErrorScreen 扩 500 态；QueryCache 全局 onError 只对"已有数据的后台刷新失败"toast；redirect 统一 pathname+search + 登录端 `safeInternalPath` 同源校验挡开放重定向。
- 验证：route-error 集成测试（渲染抛错路由→恢复 UI 而非崩溃）。

### F4 范本双轨止血
- 问题：3/4 业务页仍是被 AGENTS 判为"废弃"的横切结构，且带内联 query key（改一处另一处静默失效）；roles 守卫自称"纵向样板线"实守横切目录。
- 过程（主推止血非全迁移）：AGENTS/architecture 标注 users 为唯一纵切范本、pages/{roles,menus,dashboard} 为待迁移遗留；守卫禁止 pages/ 新增业务；`roles-vertical-standard`→`roles-page-standard`；补显式表格选型规则。
- 取舍：全量纵切迁移移入"不紧急"——成本被初稿低估（roles 主界面是 360 行权限矩阵非表格，users 流程不覆盖），风险>收益，等真实触发点。
